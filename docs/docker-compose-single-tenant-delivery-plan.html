AlphaDesk / Self-Hosted Delivery Proposal

Docker Compose 单租户交付方案

将 AlphaDesk 视为一个完整产品包交付给用户。每位用户运行自己的一套 Compose 实例： Vue 3 构建产物由 Nginx 提供，FastAPI 作为内部后端服务，WeRSS 作为固定镜像的内部信源组件。 该方案替代“共享后端多租户平台”路线，优先保留本地数据主权、简单运维和清晰的安全边界。

方案日期：2026-06-02 交付单位：一位用户 / 一套实例 运行方式：Docker Compose 默认网络：仅绑定本机 loopback

方案结论

推荐采用： AlphaDesk 不建设一个集中式共享后端，而是向每位用户交付完整的单租户 Compose 部署包。 用户只需准备 Docker Compose，执行一次启动命令，再使用浏览器访问本机 Web 入口。

重要限定： 这是“单租户自托管”方案，不等于“允许匿名暴露到公网”。默认 Compose 仅绑定 127.0.0.1。如果用户需要跨设备或公网访问，必须使用单独的 server override， 增加 HTTPS、身份认证、访问控制和备份策略。

边界变化

前一版共享后端方案试图在一个平台中承载多个用户，因此需要 workspace、对象级授权、PostgreSQL RLS 和 WeRSS runtime 调度。新方案将隔离边界前移到部署实例：每位用户拥有自己的容器组和数据目录。

领域 共享后端平台路线 Compose 单租户路线 结论
用户隔离 应用内 workspace、membership、对象级授权、数据库 RLS。 实例级隔离：不同用户运行不同 Compose project 和数据目录。 首版复杂度显著下降。
WeRSS 平台需要管理连接级状态、runtime 调度和容量配额。 每套 AlphaDesk 实例固定一个 WeRSS 容器和一份 WeRSS 数据卷。 与当前 WeRSS 单实例使用方式一致。
数据库 推荐 PostgreSQL，使用 RLS 做纵深防御。 继续使用 SQLite；一套实例只有一个数据所有者。 首版无需数据库迁移。
密钥 推荐集中式 Secret Store、KMS 和轮换机制。 本地数据卷保存 AlphaDesk Fernet key；WeRSS 密码从本实例 .env 注入。 适合本机自托管；公网服务器仍需加固。
升级 平台统一滚动升级。 用户拉取新的镜像或部署包，再执行 Compose 升级。 需要版本化发布与数据备份说明。

服务职责

服务 镜像策略 职责 持久化 默认网络
web 项目内多阶段构建：Node 构建 Vue 3，Nginx 仅保存 dist。 静态资源、SPA fallback、同源反向代理、缓存头和基础安全头。 无。 宿主机仅发布 127.0.0.1:${ALPHADESK_PORT:-8080}:80。

werss 基于固定上游 sha256 摘要构建薄运行时镜像，补齐 Playwright WebKit，不跟随 latest 漂移。 微信扫码、公众号订阅、定时更新、RSS 和管理 API。 ./data/werss:/app/data 只 expose: 8001，由 api 内部访问。
browser-worker（可选） 第二阶段按需增加，不能混入首版核心启动路径。 承载需要交互登录的 Playwright 渠道，例如知识星球。 ./data/browser-profile:/profiles 仅内部网络；调试入口必须单独鉴权。

同源路由与二维码

当前前端将 API 固定为 http://127.0.0.1:8765，后端将 WeRSS 默认固定为 http://127.0.0.1:8001。进入容器后，127.0.0.1 只表示当前容器， 因此必须改用同源路由和 Compose 服务名。

链路 容器化前 容器化后 注意事项
浏览器 → AlphaDesk API http://127.0.0.1:8765/api/... /api/...，由 Nginx 转发到 api:8765。 前端不再感知后端端口，也不再依赖宽泛 CORS。
api → WeRSS http://127.0.0.1:8001 http://werss:8001 通过环境变量覆盖默认值；数据库中的旧默认值需要迁移。
浏览器 → WeRSS 二维码 后端返回宿主机 WeRSS URL。 后端提供受控二维码代理，例如 /api/channels/wechat-mp-rss/qr-image。 不能把 http://werss:8001 返回给浏览器，也不应公开整个管理台。
浏览器 → WeRSS 管理台 直接打开 127.0.0.1:8001。 默认关闭；排障时通过运维 override 临时绑定 loopback 端口。 避免普通使用流程暴露 WeRSS 原生管理入口。 
# docker/nginx.conf 核心示意
server {
  listen 80;
  root /usr/share/nginx/html;

  location / {
    try_files $uri $uri/ /index.html;
  }

  location /api/ {
    proxy_pass http://api:8765;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Request-ID $request_id;
  }

  location = /health {
    proxy_pass http://api:8765/health;
  }
}

Compose 骨架

以下为已经落地的核心骨架。发布时由 GitHub Actions 为三个镜像生成版本标签；WeRSS 薄运行时镜像继续基于仓库当前已审核的固定摘要构建。 

name: alphadesk

services:
  web:
    build:
      context: .
      dockerfile: docker/frontend.Dockerfile
    ports:
      - "127.0.0.1:${ALPHADESK_PORT:-8080}:80"
    depends_on:
      api:
        condition: service_healthy
    restart: unless-stopped

  api:
    build:
      context: .
      dockerfile: docker/backend.Dockerfile
    environment:
      ALPHADESK_WERSS_BASE_URL: http://werss:8001
      ALPHADESK_WERSS_USERNAME: ${WERSS_USERNAME}
      ALPHADESK_WERSS_PASSWORD: ${WERSS_PASSWORD}
    expose:
      - "8765"
    volumes:
      - ./data/alphadesk:/app/data
    depends_on:
      werss:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://127.0.0.1:8765/health/ready', timeout=3)"]
      interval: 20s
      timeout: 5s
      retries: 6
    restart: unless-stopped

  werss:
    build:
      context: .
      dockerfile: docker/werss.Dockerfile
    environment:
      DB: sqlite:///data/we_mp_rss.db
      USERNAME: ${WERSS_USERNAME}
      PASSWORD: ${WERSS_PASSWORD}
      SECRET_KEY: ${WERSS_SECRET_KEY}
      GATHER.CONTENT: "True"
      GATHER.CONTENT_AUTO_CHECK: "True"
      GATHER.CONTENT_AUTO_INTERVAL: "15"
      RSS_FULL_CONTEXT: "True"
    expose:
      - "8001"
    volumes:
      - ./data/werss:/app/data
    healthcheck:
      test: ["CMD", "wget", "-q", "-O", "-", "http://127.0.0.1:8001/feed/all.rss"]
    restart: unless-stopped
不要做： 不要为了让 FastAPI 延续当前“手工启动 sidecar”接口，就把 /var/run/docker.sock 挂入 api 容器。Docker socket 基本等价于宿主机高权限控制面。 Compose 应在宿主机统一启动 WeRSS；FastAPI 只负责探测状态和返回清晰提示。

持久化、备份与升级

密钥边界： 当前 local.key 与 SQLite 同目录保存，适合作为本机数据卷中的便携式加密封装， 但不能抵御“整份数据目录被复制”的攻击。若支持公网服务器部署，应提供 Docker secrets、外部 Secret Store 或至少分离挂载目录的加固选项。

Playwright 渠道兼容性

必须单独处理： 微信公众号 WeRSS 扫码不受影响，因为二维码可以展示在 AlphaDesk Web 页面中。 但现有知识星球等渠道通过 backend/browser_session.py 启动可见 Chromium 窗口并写入 persistent profile。 FastAPI 进入 Linux 容器后，容器内浏览器无法自然弹到用户桌面。
选择 说明 适用阶段 取舍
首版暂不承诺交互式 Playwright 渠道 先交付 AkShare、BaoStock、TuShare、HTTP、MX HAR、industry-news 和 WeRSS。 MVP 推荐。 最快，但功能需要在文档中明确标记。
增加 browser-worker + noVNC 单独容器运行浏览器和 profile，通过受保护的 noVNC 页面完成登录。 需要完整功能时。 可跨平台，但增加镜像体积、鉴权、网络和运维复杂度。
宿主机浏览器桥接 由本机辅助程序完成登录和 profile 导入，再由容器采集。 桌面增强版。 用户体验更自然，但重新引入宿主机组件。

部署模式

默认：本机自托管

推荐首版 loopback only
  • 发布 127.0.0.1:8080，不向局域网暴露。
  • 用户在同一台电脑浏览器中使用。
  • 无需新增账号系统；数据目录由用户自行备份。
  • WeRSS 管理台默认不发布。

可选：私有服务器

需要加固 server override
  • 通过 Caddy、Traefik 或现有网关提供 HTTPS。
  • 增加 AlphaDesk 登录认证、速率限制和 CSRF 防护。
  • 限制 WeRSS 管理台，仅允许运维网络访问。
  • 增加定时备份、恢复演练和更新窗口。
不要混用： 本机模式可以依赖 loopback 边界；只要将端口映射从 127.0.0.1 改为 0.0.0.0，就必须按服务器模式处理。即使每套实例只有一个用户，也不能把无认证 API 直接暴露到网络。

验收矩阵

测试域 必须验证
一键启动 新目录执行初始化和 docker compose up -d 后，浏览器可访问首页，/health/ready 返回 ready。
网络暴露 默认只有 127.0.0.1:8080 可见；8765 和 8001 不发布到宿主机。
同源 API 前端所有请求使用相对路径；页面不再依赖 CORS 放行 null 或 Vite 开发端口。
WeRSS 扫码 用户可在 AlphaDesk 页面看到二维码，扫码、搜索公众号、加入订阅和采集 RSS；浏览器不访问 werss:8001。
WeRSS 隔离 两个不同 Compose project 使用不同数据目录启动后，授权、订阅和库存互不影响。
容器重建 执行 docker compose down、重新创建容器后，AlphaDesk 任务、加密配置、WeRSS 授权和订阅仍存在。
备份恢复 在新目录恢复 data 与 .env 后，健康检查、模型配置解密、WeRSS RSS 和历史快照可用。
升级 升级脚本先备份，再更新镜像；失败时可恢复前一版本镜像和数据备份。
安全 api 不挂载 Docker socket；日志不输出 API Key、WeRSS 密码、HAR 原文或 token。
旧桌面客户端清理 源码、依赖、生产启动、构建和文档不再依赖旧桌面客户端；浏览器下载 HTML 报告和打开外链正常。

待确认项

  1. 首版部署范围：是否只支持用户本机运行？建议先限定本机 loopback。
  2. 发布方式：交付源码并本地构建，还是发布固定版本的 web、api 镜像？
  3. WeRSS 管理台：是否只通过可选运维 override 暴露到 127.0.0.1:8001？建议是。
  4. Playwright 渠道：知识星球等交互式登录是否属于首版必须功能？如果是，需要增加 browser-worker 设计。
  5. 备份：首版是否提供脚本式备份恢复，还是同时提供定时任务模板？
  6. 平台化预留：未来是否仍可能建设集中式平台？若可能，当前代码应保留环境变量和服务边界，但不提前引入多租户 schema。
